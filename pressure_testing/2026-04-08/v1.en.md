### Here is what I couldn't do (not because I didn't want to, but due to constraints):
> * **Hardware Limits**: Some suggested using two separate servers. However, my current network environment cannot handle the level of throughput `gzb-one` generates.
> * **PPS Pressure**: This is a framework throughput test, not a business logic test. At 400k+ QPS, the PPS (Packets Per Second) is astronomical; standard network equipment would crash immediately. (Note: I only used two physical cores for this test).
> * **Refusal of Performance Dilution**: Network overhead only serves to mask the framework's true performance. For a test pursuing the limits of architecture, that is inherently unfair.
> * **IO Model Consistency**: The network layer of `gzb-one` is powered by Netty. At the IO level, everyone's physical limits are the same. If a framework cannot beat `gzb-one` via the "Loopback Path," there is no logical basis for it to overtake `gzb-one` in any other network environment.
> * #### Buying two high-performance servers and building a network infrastructure capable of sustaining millions of QPS just for a single benchmark is clearly irrational.

### 6. Why I didn't test the ORM
> * ORM performance testing is difficult to keep fair because my optimizations are focused on the processing workflow. For example:
> * `dao.saveAsync(entity, fail, success)`: I implemented transparent SQL merging at the low level, which makes a fair test impossible (this isn't just async; it's merging).
> * `dao.save(entity)`: If I used this framework interface, the lead would be too small—only the overhead saved by removing reflection—which isn't very interesting.
> * `dao.query(entity, sec)`: Similar reason. I provide native Query Folding, which is also "unfair."
> * If forced to use `dao.query(entity)`, the result is the same as above: the lead is too small, only saving that bit of reflection overhead. Based on my comparison with a certain framework, `find(id)` only showed a 10-20% advantage—not worth it.
> * I believe high-frequency interfaces SHOULD use `saveAsync` and `dao.query(entity, sec)`. If I'm forced to choose a sub-optimal solution just for the sake of a test, I’d rather not test it at all.

### 7. Why is there such a massive difference in execution efficiency?
> * **Dynamic Weaving**: Take decorators (Controller AOP) for example. `gzb-one` only inserts decorator invocation code when it detects you actually need it. If you don't, there isn't even a single `if` statement.
> * **Flat Processing**: You can observe the overhead of `gzb-one` via the Flame Graph: [gzb-one Flame Graph](https://qq1297368889.github.io/gzb_java/pressure_testing/2026-04-08/gzb_one.html)
> * **Compile-time Enhancement**: For certain processing and logic checks, `gzb-one` chooses to handle them during compilation. The core execution path is a "straight highway."
> * **Zero Reflection**: The core path is reflection-free, relying entirely on compile-time enhancement. During startup or when **Seamless Hot Updates** are detected, the framework uses compile-time enhancement to process your classes.
> * **Memory Pooling**: This wasn't actually reflected in the stress test because I didn't use my own serialization implementation. I used a third-party library because that library is ridiculously fast and surpassing it would be difficult. I was lazy...
> * The truth is, it’s not about superior or inferior code; it’s an architectural generational gap. I haven't even extremely optimized the framework yet; I already know parts that aren't optimized where at least 100 nanoseconds could be saved per request.

### 8. Framework Completeness Statement
> * `gzb-one` is not a benchmarking toy. It is fully featured with outstanding performance. Its only minor shortcoming is that it has only been through a small number of real-world web projects.
> * **Zero-Dependency Execution**: No need to import bloated external libraries. It is a high-performance universe unto itself, naturally immune to dependency conflict issues.
> * **Full-Stack Capability**: Built-in IOC, AOP, ORM, Cache, Log, and Hot Update.
> * **Seamless Hot Updates**: Supports **Dynamic Logic Hot-Reloading** under extreme high-QPS loads with no significant QPS fluctuations.
> * **Low-Code Engine**: Built-in lightweight driver (reverse-generates: entity, dao, action, web ui based on database info), balancing extreme performance with development efficiency.
> * **Multi-Protocol Unification**: A single Action can simultaneously serve HTTP / TCP / UDP (UDP Demo is already working).
> * These are just some of the features. For details, refer to: [Open Source Repository Documentation](https://github.com/qq1297368889/gzb_java/blob/main/doc.md)

## gzb-one: Thoughts on Performance and Trade-offs
### Core Philosophy: Travel Light, Explode with Power
As pioneers, Spring and Quarkus carry too much **historical baggage** to maintain massive ecosystem compatibility; **gzb-one**, however, chooses to **travel light** in 2026.
> **Their goal is to be compatible with everything; my philosophy is to be just a bit faster.**

---

### ⚖️ Honest Trade-offs
In architectural design, there is no such thing as a free lunch. We must honestly face the trade-offs:
> * Different Pursuits: Performance is not their only goal, just as extreme compatibility is not my forte.
> * Proactive Trimming: To squeeze out every drop of hardware performance, I proactively sacrificed some "abstraction-for-the-sake-of-abstraction" generality.
> * Cognitive Difference: From the universal perspective of Java specifications, I seem to have "sacrificed" a lot; but from the perspective of real-world backend development, I believe I have sacrificed almost nothing.
> * Is it worth it? The answer is a resounding YES. In a production environment where every millisecond counts, this "sacrifice" is the only path to the extreme.

### Test Code:
> * Below is the code for **gzb-one**. The code for all other frameworks is identical, with the exception of different annotation rules.
> * I am not an optimization expert for every framework, but I did not restrict any of their running parameters.
> * All frameworks were started with default parameters; the startup command was a simple `./xxxx` with no additional arguments.

```java
package gzb.start;
import gzb.frame.annotation.*;
import gzb.frame.factory.ContentType;
import gzb.tools.Config;
import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.ThreadLocalRandom;
@Controller
@RequestMapping("test")
@Header(item = {@HeaderItem(key = "Content-Type", val = ContentType.json)})
public class Test {
    public static final byte[] HELLO_WORD = "Hello, World!".getBytes(Config.encoding);
    @EventLoop
    @RequestMapping("hello")
    @Header(item = {@HeaderItem(key = "Content-Type", val = ContentType.html)})
    public byte[] hello() {
        return HELLO_WORD;
    }
    @EventLoop
    @RequestMapping("users")
    public Users users(Long usersId, String usersName, String usersPassword, String usersEmail, Integer usersAge) {
        int ram_id = ThreadLocalRandom.current().nextInt(0, 10);
        return new Users(
                usersId == null ? (long) ram_id : usersId + ram_id,
                usersName == null ? "vpeGb2SNo8Xk" : usersName,
                usersPassword == null ? "az1Amb2aVJsP" : usersPassword,
                usersEmail == null ? "nWV3qO6qW0zZ@gzb.com" : usersEmail,
                usersAge == null ? 12 : usersAge
        );
    }
}
```
